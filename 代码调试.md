# PEARL 维护决策 RL 代码调试记录

## 1. 环境设计演进

### 1.1 原始设计（8 动作）
- "运行"是 agent 可选动作之一，agent 可以选择永远不跑设备
- 8 个动作：run, repair_A, repair_B, repair_AB, corrective_A, corrective_B, corrective_AB, replace
- **根本问题**：action space 结构允许"不运行"，agent 可以通过不执行 run 来规避所有风险

### 1.2 重设计后（5 动作，当前版本）
- 每步设备**自动运行**（pointer 自动推进），agent 只决定是否维修
- 5 个动作：0=不干预, 1=修A, 2=修B, 3=修AB, 4=更换
- 删除 corrective repair（与 preventive 功能重复）
- 从结构上消除了"不跑"的 exploit

## 2. 已修复的 Bug

### 2.1 单 pointer bug
- **问题**：原始代码用 `pointer = max(pA, pB)` 统一管理两个部件，repair A/B 无法独立降低各自 pointer
- **修复**：改为双独立 pointer（pointer_A, pointer_B），各自独立退化和维修

### 2.2 `_restore_point` 推高 pointer
- **问题**：`pointer = restore_point(count)` 在 restore_point > 当前 pointer 时反而把 pointer 推高
- **修复**：改为 `pointer = min(pointer, restore_point(count))`，确保维修只能降低 pointer

### 2.3 best_model 保存被误删
- **问题**：移除 collapse recovery 代码时连带删除了 best_model 保存逻辑
- **修复**：加回 best_model 保存（仅保存，不做 restore/LR 修改）

### 2.4 over_repair exploit（-50 < -200）
- **问题**：agent 学会连续维修 15 次触发 over_repair penalty（-50），因为比跑到故障（-200）更便宜
- **修复**：over_repair penalty 从 -50 提升到 -100（自动运行版本）

### 2.5 action 4 (replace) 绕过 over_repair
- **问题**：replace 动作将 repair_in_row 重置为 0，agent 用 replace 插入来绕过 over_repair 计数
- **修复**：在自动运行版本中 replace 也计入 repair_in_row（不再重置）

### 2.6 step_reward 变负
- **问题**：`step_reward` 在高退化进度时变成负值，`reward -= step_reward` 反而成了补贴
- **修复**：自动运行版本中 R_run = 3.0 恒正，无条件获得

## 3. 参数调优历程

### 3.1 C_prev 过高（8.0 → 3.0）
- C_prev=8.0 时，第一次维修成本 = 8 × (1 + 0.3×1) = 10.4，远高于单步收益 3.0
- agent 直接放弃维修，纯跑 action 0
- 降到 C_prev=3.0 后训练稳定，不再崩溃

### 3.2 entropy_coef 过高（0.15 → 0.08）
- entropy_coef=0.15 导致 agent 无法收敛，持续随机探索 action 4 (replace)，污染 critic
- 降到 0.08 后恢复稳定

### 3.3 参数扫描结果（sweep_entropy.py）

三组配置在 800 episode 训练后的对比：

| 配置 | 最佳 reward | 最后 50 ep 均值 | 正 reward 回合 | Action 0 占比 >80% |
|------|-----------|---------------|--------------|-------------------|
| entropy_coef=0.05, init_temp=1.4, decay=300 | **10.0** (ep 13) | -352.8 | **10/800** | 319/800 |
| entropy_coef=0.08, init_temp=2.0, decay=600 | 6.1 (ep 13) | -793.7 | 2/800 | 218/800 |
| entropy_coef=0.10, init_temp=2.0, decay=600 | -29.9 (ep 16) | -1380.0 | 0/800 | 12/800 |

**结论**：
- entropy_coef=0.05 最佳，但峰值仅 10.0，远低于理论最优 +600
- entropy 越高越差：0.10 时策略完全发散，大量 action 4 (replace) 导致 -3000 级负收益
- 所有配置在训练后期都不稳定，reward 没有收敛趋势

## 5. 理论策略收益分析（seq_len ≈ 76）

| 策略 | 总 reward | 说明 |
|------|----------|------|
| 纯 run（不修） | 75×3 - 200 = **+25** | 跑到头吃故障 penalty |
| run + 2-3 次战略维修 + 存活 | 200×3 - ~50 + 50 = **+600** | 最优策略 |
| 纯修 AB × 10 → over_repair | 10×(3 - 12~) - 100 ≈ **-190** | 连续维修自惩罚 |
| 纯 replace × 10 | 10×(3 - 40) - 100 = **-470** | 最差策略 |

## 6. 当前核心问题

**agent 过早收敛到纯 action 0（不干预），无法发现维修的长期价值。**

- 训练前 10 episode，entropy 从 1.6 迅速降到 < 0.1
- agent 快速学会"不修"可以拿 +25（纯 run 到故障），然后锁死在这个局部最优
- 维修的长期收益（延长寿命 → 多拿未来 R_run）需要跨越数十步的信用分配，critic 无法学到

**环境设计已确认正确**，问题在训练侧（探索-利用平衡、信用分配）。

## 7. 测试结果现状

当前 best model 在测试集上的表现：
- **Unit 14**: 300 回合全部 -467 ~ -470，几乎全是 action 4 (replace) ×10，被困在最差策略
- **Unit 15**: 300 回合全部 -470，同上

说明 best_model 保存的是训练早期偶然拿到正 reward 的 snapshot，但该模型并未真正学会维修策略。

## 8. 关键教训

1. **reward shaping 不能解决结构问题**：反复调 penalty/cost 数值是 band-aid，需要从 action space 结构入手（自动运行设计）
2. **entropy 调参有上界**：entropy 过高（>0.10）导致 action 4 (replace) 被频繁采样，-40 的成本污染 critic，训练崩溃
3. **局部最优陷阱**：纯 action 0 给 +25 reward，是一个强吸引子。agent 需要足够的探索才能发现维修的长期价值（+600），但过多探索又会触发高成本动作
4. **信用分配困难**：维修的收益是"100 步后不故障"，这种长程依赖对 single-step TD learning 很困难

## 11. 多种子诊断：定位 auto-alpha 为根因（2026-06-29）

> 架构已从纯 entropy-bonus 演进为 **discrete SAC + auto-entropy tuning**（critic 每步更新，
> actor loss = `(probs * (alpha*logπ - Q)).sum()`，alpha 由 `log_alpha` 自动调温，
> `target_entropy = 0.5·ln(5) ≈ 0.805`）。仍然不收敛，遂做诊断。

### 11.1 调试方法

1. **环境二次确认（diagnose_env.py，只读不训练）**：把每步 reward 精确拆解为
   `run + cost + shaping + terminal`（cost 由反推得到，不复制成本公式）。
   - 实验A：阈值策略可达 **+633**，理论最优可达。
   - 实验B：shaping_k 0→60 扫描，**不改变策略排名**（证伪"shaping 鼓励 spam"假设）。
   - 实验C：单步 shaping ≈ -0.5/step，尺度无害。
   - **结论：环境正确，问题 100% 在训练侧。**

2. **诊断探针（pearl.py → diagnostics.csv）**：固定参考态 `s_ref`，逐 episode 记录
   `reward, alpha, ent_ref, argmax, p0..p4, q0..q4, z_norm, loss_actor, loss_critic`，跨 episode 可比。

3. **多种子实验（新增 `--seed` 开关）**：同一 config 跑 seed 0~5（各 300 ep），
   日志分别落到 `logs/compare/PEARL_seed{N}/`，用于区分"运气/种子依赖" vs "系统性必然"。

### 11.2 多种子结果

| seed | 崩溃ep(ent<0.1) | 锁定动作 | 曾达到的最高 reward | **alpha 终值** | 最终 reward |
|------|----------------|---------|-------------------|--------------|------------|
| 0 | 29 | 1 修A | 从未为正 | **0.27** | -154 |
| 1 | 50 | 2 修B | 从未为正 | **0.27** | -154 |
| 4 | 35 | 1 修A | 从未为正 | **0.28** | -154 |
| 5 | 43 | 1 修A | 从未为正 | **0.25** | -154 |
| 2 | 39 | 0 run | **+445 @ep103** | **7.0** 💥 | -155 |
| 3 | 11 | 0 run | **+322 @ep43** | **5.0** 💥 | -133 |

**alpha 终值分两组**，正是答案所在。

### 11.3 两种失败模式（根因同一个）

**模式A — 死锁（seed 0/1/4/5）**：早早锁死到 *修A/修B*。修A 连修触发 over_repair 提前终止
→ episode 很短 → 每 episode 梯度更新次数少 → **alpha 积分太慢**（240 ep 才从 0.2 爬到 0.28），
来不及把熵拉回来 → 熵=0、reward=-154 永久卡死。

**模式B — alpha 风暴震荡（seed 2/3）**：运气好锁到 *动作0 run*（不错的策略，一度冲到 **+445**）。
但 run 能活满 200 步 → 每 episode 更新次数巨多 → **alpha 积分爆炸**：0.2→0.5→1.5→2.8→**7.0**。
alpha>4 后熵项 `alpha·logπ`(≈11) 压过 Q(≈50)，actor 被强行推回均匀分布
（seed2: p0 从 1.0 掉到 0.59），**亲手撕碎 +445 的好策略** → reward 崩回 -200~-800，
且 alpha 卡在 7 下不来（无 anti-windup，恢复需上千 ep）。

### 11.4 根因结论

1. **"replay buffer 单向死循环"假设被证伪**：seed2/3 证明策略能逃出（冲到 +445）、
   熵能自恢复（0→1.1），buffer 不是单向陷阱。
2. **真正根因 = auto-alpha 是个没有 anti-windup 的积分控制器，且积分速率被 episode 长度绑架**：
   短 episode → 太慢救不回（模式A）；长 episode → 爆到 7 毁掉好策略（模式B）。
   从未在 `target_entropy≈0.805` 处稳定过。
3. **种子只决定"先锁到哪个动作 / 有没有尝过甜头"**——所以光换种子永远不稳。

### 11.5 否定的修复方向

- ❌ **`entropy_coef: 0.2 → 5.0`**：代码 [pearl.py:139](src/pearl.py#L139) 中
  `log_alpha = log(entropy_coef)`，**entropy_coef 只是 alpha 的初始值**，auto-tuning
  ([pearl.py:302-305](src/pearl.py#L302-L305)) 随即接管。改成 5.0 不解决失控控制器，
  反而让 alpha 从第 1 个 episode 就空降到崩坏区（数据已证 alpha≈5~7 时 reward 恒为 -150）。

### 11.6 建议修复（两件事，缺一不可）

1. **干掉/驯服 auto-alpha（根因）**：关掉自动调温（[pearl.py:302-305](src/pearl.py#L302-L305)），
   改用**固定衰减温度** `alpha: 0.5 → 0.02`（复用 `temp_decay_ep`）。
   早期 0.5 压住模式A的过早死锁；晚期 0.02 允许 commit 到确定性好策略，消灭模式B爆炸。
2. **demo/阈值策略种子注入**：用能拿 +633 的阈值策略预填 replay buffer，
   让每个种子都稳定见到好轨迹，不再靠运气撞上 run。

### 11.7 本轮调试新增/改动文件

- `experiments/diagnose_env.py`（新）— 环境 reward 拆解验证（只读）
- `src/pearl.py` — 新增诊断探针写 `diagnostics.csv`；新增 `set_seed()` 与 `cfg["seed"]` 读取

## 12. 第二轮修复尝试：调 alpha 初始值 + target_entropy（2026-06-29）

### 12.1 修改内容

基于 11.6 的分析方向 1，尝试不去掉 auto-alpha，而是调整其两个参数：

| 参数 | 修改前 | 修改后 | 理由 |
|------|--------|--------|------|
| `entropy_coef`（alpha 初值）| 0.2 | 2.0 | 防止前 30 集 Q gap 3 → exp(3/0.2)=10^6 直接杀死探索 |
| `target_entropy` | 0.5·ln(5)=0.805 | 0.1·ln(5)=0.161 | 最优策略近确定性，target=0.8 导致 alpha 停在 5~7 不下来 |

文件改动：
- `configs/pearl_default.yaml:29` — `entropy_coef: 0.2 → 2.0`
- `src/pearl.py:187` — `target_entropy = 0.5 * np.log(n_actions) → 0.1 * np.log(n_actions)`

### 12.2 实验结果（seed 0, seed 2，各 300 ep）

**两个 seed 全部崩溃**，与之前一模一样锁死单一动作。

| seed | 锁死动作 | 最终 reward | alpha 轨迹 | 测试 unit14 |
|------|---------|------------|-----------|------------|
| 0 | action 2 ×10 | -154 | 2.0 → 1.40(ep29) → 1.47(ep300) | -153 |
| 2 | action 1 ×10 | -154 | 2.0 → 1.64(ep29) → 1.44(ep300) | -153 |

### 12.3 失败原因：auto-alpha 方向反了

Alpha 在前 30 集从 2.0 **掉到 1.4**，而不是维持或上升：

```
auto-alpha 梯度 = entropy - target_entropy
前几集策略随机: entropy ≈ 1.6, target = 0.161
梯度 = 1.6 - 0.161 = +1.44 (大正值)
→ log_alpha 增大 → alpha 上升? 不！
loss_alpha = log_alpha * (entropy - target) → 正值
梯度下降 minimize loss → log_alpha 减小 → alpha 下降!
```

**Auto-alpha 在策略随机（最需要探索）时把 alpha 往下压**，因为它看到 entropy > target，认为"探索够了，该降温"。等策略真的崩到 entropy ≈ 0 后，梯度 = 0 - 0.161 = -0.16（绝对值小 10 倍），alpha 缓慢回升但为时已晚——Q gap 已达 30+，exp(30/1.4)=10^9。

**结论：auto-alpha 机制在此环境有结构性缺陷，无论怎么调参（初始值、target）都无法修复。**

### 12.4 最终根因定位：Critic Q-gap 自我放大

回看所有失败 seed 的 Q 值演化，发现 **alpha 不是根因，critic 才是**：

```
Seed 0 ep29:  Q = [-118.8, -119.7, -117.1, -118.8, -120.1]  gap=3
              alpha=1.4, exp(3/1.4)=8.5, p(a2)=53%  ← 策略还活着

Seed 0 ep59:  Q = [-174.9, -161.7, -143.5, -167.8, -167.5]  gap=31
              alpha=1.39, exp(31/1.4)=10^9, p(a2)=99.99%  ← 死了
```

**30 集内 Q gap 从 3 爆炸到 31。** 即使 alpha=10，exp(31/10)=22，后续 Q gap 会继续涨到 60、100，任何 alpha 值都扛不住。

**Critic Q-gap 自我放大的死循环：**

1. Critic loss 只更新被选动作的 Q 值：
   `q_a = critic(bs, z_b).gather(1, ba)` → `loss_c = smooth_l1(q_a, td_target)`
2. 当 replay 70% 是 action 2 时：
   - Q(a2) 被准确拟合到真实回报 ≈ -153
   - Q(a0), Q(a3) 等没有直接梯度信号，被共享权重拖拽到 -175（偏低 22 点）
3. Discrete SAC actor 用全部 Q 值决策：`loss_a = (π * (α·logπ - Q_all)).sum()`
   - 看到 Q(a2)=-143 > Q(a0)=-175 → 选 a2 概率↑
   - 更多 a2 数据 → Q gap 更大 → 完全锁死
4. **Alpha 无论多大都无法打破此循环**——因为 Q gap 增长速度（30/30ep ≈ 1/ep）
   远超 alpha 的调节能力

**核心：critic 对未见动作给出错误的低 Q 值，discrete SAC 信以为真。**

### 12.5 修复方向变更

之前方向（调 alpha / alpha schedule）被证伪。新方向：**给 critic 提供所有动作的真实数据**。

用户已在代码中实现 Demo 注入机制（[pearl.py:95-138](src/pearl.py#L95-L138)）：
- `_threshold_policy()`：阈值专家策略，pointer 超过 thr×寿命就修，否则 action 0
- `seed_demonstrations()`：用阈值专家生成高回报轨迹预填 replay buffer
- 阈值抖动 `thr ∈ [0.5, 0.85]`，制造多样的好轨迹（包含所有动作类型）
- 配置项：`pearl_default.yaml` 新增 `demo_episodes: 30`

**预期效果**：critic 从第 1 集就看到 reward≈+600 的轨迹，包含 action 0/1/2/3 的合理分布，
Q 值初始化在正确量级 → Q gap 反映真实价值差异而非数据偏差 → 打破死循环。

### 13.2 诊断 train/test gap（diagnose_policy.py，只读）

新增 [experiments/diagnose_policy.py](experiments/diagnose_policy.py)：加载 best_model，纯 run 把 pointer 一路推高，
逐步打印 actor 的 π 与 critic 的 Q。发现 **actor 完全无辜，问题在 critic**：

1. **actor 忠实跟随 critic**：argmax 永远 = Q 最大的动作。
2. **critic 学成状态-动作可分离的错误 Q**：`Q(s,a) ≈ f(pointer) − g(a)`，动作偏好 `g(a)` 固定不随
   pointer 变化（run 永远最优，维修一律打折 ~10）。`maxRepairQ − runQ` 几乎全程 ≈ **−10**，
   只在 pointer>0.9L 最后 2~3 步才微微转正，且转正的是"更换"（最贵）。
3. **深层根因 = critic 严重高估纯 run**：t=10 时 Q(run)=124，但真实纯 run 只值 +6.6。
   原因：**buffer 缺"该修不修→撞故障 -200"的负样本**。demo 专家与训练策略都在故障前维修，
   critic 从没见过纯 run 的下场，于是默认 run 永远安全。

### 13.3 概念辨析：是 OOD 高估，不是信用分配

| 概念 | 定义 | 本案 |
|------|------|------|
| 信用分配 credit assignment | 延迟 reward 难以归因到很久前导致它的动作 | 次要/潜在 |
| **分布外高估 OOD / extrapolation error** | critic 对训练数据没覆盖的 (s,a) 外推出错误（偏高）Q | **主因之一** |
| **bootstrap 乐观偏差** | TD target 用 `max_a' Q(s')` 假设后继最优，但贪婪策略实际不那样走 | **主因之二** |

不是经典信用分配（前提是 buffer 里有负轨迹但归因失败），而是 **buffer 根本没有负轨迹 → critic 瞎外推**。
判别实验：注入负样本后看 Q(run) 是否被压下去——**压下去=OOD 覆盖问题（确认）；仍高=才是信用分配**。

### 13.4 阶段二：负样本注入（纯 run 跑到故障）

重构 `seed_demonstrations`，新增 `demo_neg_episodes: 15` 条**纯 run 一路跑到故障**的轨迹
（`policy_fn = lambda env: 0`，绝不维修）。最后一步带 -200 penalty，`n_step=20` 把它传播回
故障前 20 步的高 pointer run 动作 → critic 第一次见到"高 pointer + run"区域的真实坏后果。

### 13.5 判决结果：OOD 高估确认，条件维修学会

**Q(run) 在高 pointer 被专门压下，甚至转负**（判别实验通过）：

| `maxRepairQ − runQ` | 注入前 | 负样本注入后（seed2 unit18）|
|---|---|---|
| pointer ~0.8L | ≈ **−4**（run 仍最优）| **+11.7**（维修胜出）|
| pointer 极限 | +1~+4 | **+22.9**，Q(run) 跌到 **−8.6** |

**argmax 提前切换维修**：seed0 切换点 t≈0.87L（注入前 0.93L），seed2 t≈0.82L。两 seed 在 train units 上都学会条件维修。

**测试起飞**：seed2 unit14 出现 **+576 × 248 次**（接近理论最优 +600）+ 199×19 + 6.6×33；
seed0 unit15 混合 173~556。训练 reward +480~+518。

### 13.6 残留两问题（下一轮目标）

- **问题 A — 切换偏晚 + 切错动作**：理论最优应 pointer ~0.5-0.6L 战略修 A/B（成本 3），
  现在拖到 ~0.82-0.89L 才动，且 argmax 切的是"更换"（成本 40）。高 pointer 下
  `Q(更换)>Q(修AB)>Q(修A)`，critic 认为更换续命最多但没算清"修 A/B 更划算"。是 +576 vs +600 的差距来源。
- **问题 B — 泛化不稳，依赖 z 采样**：seed2 unit14 优秀但仍有 33 次掉回纯 run；**seed0 unit14 测试仍全 6.6**
  （train units 学会了维修，泛化到 test unit14 失败）。测试 z 采样带噪声，策略时好时坏，方差大。
  属过拟合/泛化层面，与已修好的 critic 病理是两件事。

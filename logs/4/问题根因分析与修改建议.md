# 测试时动作2主导问题：根因分析与修改建议

## 一、问题现象

- 测试时 **action 2（提前维修 B）** 出现 12~26 次/episode，占主导
- **action 0（继续运行）** 仅 0~12 次，多数 episode 中偏少
- 真实场景期望：**健康时以运行为主**，action 0 应最多

---

## 二、全链路数据流梳理

```
原始 CSV (cycle + 传感器)
    ↓ step4.2 质量增强
滑动窗口 (seq_len=30) → sequences_complete.npy: (N, 30, 172)
    ↓ load_sequences
每个样本 = 一个 30 步窗口 (30, 172)
    ↓ env: pointer 0→29, encode_state = sequence[:pointer+1]
GRU 编码 prefix → state (64 维)
    ↓ Actor(state, z)
动作概率 → 采样动作
```

---

## 三、根因分析

### 1. 【核心】GRU 未训练，state 无健康语义

**位置**：`pearl.py:50-54`

```python
def load_real_gru_encoder():
    model = GRUEncoder().to(DEVICE)
    print(">> GRU 模型已加载（随机初始化，若有预训练模型可替换加载）")
    return model
```

- GRU **随机初始化**，从未训练
- state 本质是随机投影，**不包含健康/退化信息**
- 策略无法学到「健康 → action 0」「退化 → action 2」的映射

### 2. 数据形式与 env 假设不一致

**数据处理**（`step4.2-质量增强与序列构建.py:82-95`）：

- 使用 **滑动窗口**：`for end in range(seq_len-1, T)`
- 每个序列 = 30 步窗口，形状 `(30, 172)`
- 不同窗口来自不同时间片段，彼此独立

**环境假设**：

- `pointer` 表示「在退化轨迹中的位置」
- `pointer=0` 视为健康，`pointer→seq_len` 视为退化
- 实际：每个序列只是 30 步局部窗口，`pointer` 只表示「窗口内位置」，不表示「全寿命位置」

**结果**：env 的「健康/退化」语义与数据不对齐。

### 3. 动作 A/B 与数据无对应关系

- 数据只有传感器序列，没有「部件 A」「部件 B」标签
- action 1/2/3 在数据层面完全对称
- 策略对 1、2、3 的偏好主要来自随机初始化和早期探索，易塌缩到某一动作（如 action 2）

### 4. 测试时 context 为空

**位置**：`pearl.py:416-421`

```python
ctx = ctxbuf.sample_context()
if ctx is None:
    ctx_t = torch.zeros((1, context_input_dim), device=DEVICE)
```

- 测试开始时 context buffer 为空
- `z` 从零向量附近采样，与训练时的 z 分布不一致
- 策略在「陌生 z」下表现不稳定，易偏向训练中常见的动作（如 action 2）

### 5. 奖励结构对 action 0 激励不足

| 动作 | 奖励 | 说明 |
|------|------|------|
| 0 | +10 | 固定 |
| 2 | +7→+1 | 首次 +7，多次递减 |

- 单步差距仅 3，且 action 2 会重置 pointer，带来「可重复选择」的错觉
- 策略容易在早期探索中偏向 action 2，并形成正反馈

---

## 四、修改建议（按优先级）

### 1. 【高】在 state 中显式加入位置信息

**目的**：让策略知道「当前在轨迹中的位置」，从而区分健康/退化。

**修改**：`env.py` 的 `encode_state()`

```python
def encode_state(self):
    # ... 原有 GRU 编码 ...
    emb = self.encoder(seq_tensor).squeeze(0).cpu().numpy()  # (64,)
    # 追加归一化位置：pointer/seq_len
    pos = np.array([self.pointer / max(1, self.seq_len)], dtype=np.float32)
    return np.concatenate([emb, pos])  # (65,)
```

**配套**：`pearl.py` 中 `state_dim` 改为 65，`context_input_dim` 相应调整。

**效果**：策略可学到「pos 小 → 多选 action 0」「pos 大 → 多选维修」。

---

### 2. 【高】使用完整退化轨迹替代滑动窗口

**目的**：让 `pointer` 真正表示「全寿命中的位置」。

**修改**：在 `step4.2` 中增加「完整轨迹」输出，例如：

```python
# 除滑动窗口外，再保存完整轨迹
def save_full_trajectory(df_prepared, feature_cols, save_path):
    T = len(df_prepared)
    trajectory = []
    for t in range(T):
        row = df_prepared.iloc[t]
        v = []
        for f in feature_cols:
            v.extend([row[f+'_imputed'], row[f+'_mask'], row[f+'_delta'], row[f+'_outlier']])
        trajectory.append(v)
    np.save(save_path, np.array(trajectory, dtype=np.float32))  # (T, 172)
```

**修改**：`load_sequences` 改为加载 `trajectory_complete.npy`，每个 unit 一个序列 `(T, 172)`。

**效果**：`pointer` 与真实退化阶段一一对应，更符合 env 设计。

---

### 3. 【高】预训练 GRU 编码健康/退化

**目的**：让 state 具有健康语义。

**做法**：

- 用 RUL 或健康指标做监督（若数据中有）
- 或做自监督：预测下一时间步、重构等
- 训练完成后，固定或微调 GRU，再用于 PEARL

**效果**：state 能反映健康程度，策略可据此选择 action 0 或维修。

---

### 4. 【中】强化 action 0 的奖励

**修改**：`env.py`

```python
self.task_reward = 15  # 从 10 提高到 15
# 或：健康时额外奖励
# if self.pointer / self.seq_len < 0.3:  # 前 30% 视为健康
#     reward += 5
```

**效果**：提高「继续运行」的吸引力，抑制过度维修。

---

### 5. 【中】测试时使用确定性策略

**修改**：`pearl.py` 的 `evaluate()`

```python
# 将 action = np.random.choice(n_actions, p=probs)
# 改为
action = np.argmax(probs)
```

**效果**：评估时去掉随机性，更稳定地反映策略真实偏好。

---

### 6. 【中】测试前预填充 context

**修改**：`evaluate()` 开始时，用若干步「虚拟」或真实 rollout 填充 context buffer，再开始正式评估。

**效果**：测试时的 z 分布更接近训练，减少因 context 为空导致的偏差。

---

### 7. 【低】端到端训练 GRU

**目的**：让 GRU 直接为决策任务服务。

**做法**：将 GRU 移入 agent，参与 actor/critic 的梯度更新（可对 GRU 使用较小学习率或部分层冻结）。

**效果**：state 更贴合决策需求，但实现和调参成本较高。

---

## 五、建议实施顺序

1. **先做**：在 state 中加 `pointer/seq_len`（改动小、见效快）
2. **再做**：调整数据为完整轨迹，并修改 `load_sequences`
3. **视情况**：预训练 GRU、提高 `task_reward`、测试时用 argmax
4. **有余力**：测试前预填充 context、端到端训练 GRU

---

## 六、小结

| 根因 | 影响 | 建议 |
|------|------|------|
| GRU 随机初始化 | state 无健康语义 | 预训练 GRU 或加位置特征 |
| 滑动窗口 vs 完整轨迹 | pointer 语义不准 | 使用完整退化轨迹 |
| A/B 无数据对应 | 动作偏好随机 | 加位置特征，强化 action 0 |
| 测试 context 为空 | z 分布偏移 | 预填充 context 或 argmax 评估 |
| 奖励差距小 | action 0 激励不足 | 提高 task_reward |

核心结论：**state 缺乏可区分的健康信息**，是 action 2 主导的主要原因。优先在 state 中加入 `pointer/seq_len` 或改用完整轨迹，可显著改善「健康时多运行」的行为。
